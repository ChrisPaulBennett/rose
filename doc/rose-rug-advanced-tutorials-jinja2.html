Rose Advanced Tutorial: Suite Date/Time Manipulation

Introduction

This part of the Rose user guide walks you through using templating and rose-suite.conf variables in your suite.rc, using Jinja2.

This allows you to collapse repeated configuration, and move commonly used settings to a central place. It is particularly useful for suites with many, very similar tasks.

Example

We'll demonstrate templating by using the finale of a firework display as an example.

Firework displays, if done right, have sophisticated time-sensitive scheduling with a large number of similar 'tasks' - ignitions.

This suite.rc has two families of tasks - tasks within a family are either identical (IGNITION family) or slightly different ( DETONATE family). There are clear patterns in both the dependency graph and the [runtime] section.

We want to collapse most of this down with Jinja2.

For Loops

Let's have a look at using a for loop in Jinja2.

The dependency graph for the ignition tasks follows a simple pattern for the first 16 tasks. We can replace the lines from:

            ignite_rocket_00 => \

to

ignite_rocket_15 => \

with:

For Loop, Part 1

{%- for num in range(16) %}
           ignite_rocket_{{ num }} => \
{%- end for %}

This uses Jinja2 blocks ({% to %}) to template the ignite_rocket... line 16 times, subsituting a number rocket_num in the line each time ({{ to }}).

This will produce something like:

            ignite_rocket_0 => \
            ignite_rocket_1 => \
            ignite_rocket_2 => \
            ignite_rocket_3 => \
            ...

Variable Assignment

Our original text had properly formatted number suffixes that would sort correctly in cylc gui (_00, _01).

We can produce nicely formatted numbers by creating another variable in Jinja2, inside the for loop. Replace the for loop text with:

{%- for num in range(16) %}
{% set num_label = '%02d' % num %}
           ignite_rocket_{{ num_label }} => \
{%- end for %}

Whitespace Control

This would produce output like this:

           ignite_rocket_00 => \
           
           ignite_rocket_01 => \
           
           ignite_rocket_02 => \

which is nearly correct! We forgot to add the - after {%, which removes the newline for the set num_label line. Add it in.

For loop, part 2

We can template away the rest of the graph in exactly the same way. Replace ignite_rocket_16 & \ to ignite_rocket_28 with:

{%- for num in range(16, 29) %}
{%- set num_label = '%02d' % num %}
                          ignite_rocket_{{ num_label }} & \
{%- end for %}

However, this doesn't handle the special case of the ignite_rocket_28 => \ line, and it feels redundant to have an almost duplicated loop below our first one.

If Blocks

Jinja2 supports if blocks, so we can actually change what we do based on the value of a Jinja2 variable. Replace the first and second loops with:

{%- for num in range(30) %}
{%- set num_label = '%02d' % num %}
{%- if num <= 15 or num == 29 %}
            ignite_rocket_{{ num_label }} => \
{%- else %}
                          ignite_rocket_{{ num_label }} & \
{%- endif %}
{%- endfor %}

This will give us the correct output, barring some indentation for ignite_rocket_29 => \.

For Loops, part 3

We can also trivially replace the last part of the dependency graph. Replace the whole ignite_rocket_00 => detonate_rocket_00 to ignite_rocket_29 => detonate_rocket_29 loop with:

{%- for num in range(30) %}
{%- set num_label = '%02d' % num %}
            ignite_rocket_{{ num_label }} => detonate_rocket_{{ num_label }}
{%- endfor %}

Finished graph

Your dependency graph configuration should look like:

        graph = """
            start => \
{%- for num in range(30) %}
{%- set num_label = '%02d' % num %}
{%- if num <= 15 or num == 29 %}
            ignite_rocket_{{ num_label }} => \
{%- else %}
                          ignite_rocket_{{ num_label }} & \
{%- endif %}
{%- endfor %}
                                        stop
{%- for num in range(30) %}
{%- set num_label = '%02d' % num %}
            ignite_rocket_{{ num_label }} => detonate_rocket_{{ num_label }}
{%- endfor %}

Runtime Loops

We can also replace some of the runtime - the ignite_rocket_ tasks are extremely repetitive (and therefore easy!), but the detonate_rocket_ tasks differ from task to task. We'll start with the ignite_rocket_ tasks - replace the sections ignite_rocket_00 to ignite_rocket_29 with:

{%- for num in range(30) %}       
{%- set num_label = '%02d' % num %}
    [[ignite_rocket_{{ num_label }}]]
        inherit = IGNITION
{%- endfor %}

Runtime Loops - Patterns

The detonate_rocket tasks are more complex, but have some repeating patterns based on the task number - SOUND alternates, and COLOUR_CODE repeatedly loops from 1 to 5.

This means we can extract this information from the task number with some Jinja2 magic.

Runtime Loops - Replacement

Replace the section detonate_rocket_00 to detonate_rocket_29 with:

{%- for num in range(30) %}
{%- set num_label = '%02d' % num %}
    [[detonate_rocket_{{ num_label }}]]
        inherit = DETONATE
        [[[environment]]]
            COLOUR_CODE = {{ num % 5 + 1 }}
            SOUND = {{ ["BANG", "whoosh"][rocket_num % 2] }}
{%- endfor %}

Runtime Loops - Explanation

This uses the remainder operator of Jinja2/Python (%) to extract the correct number for COLOUR_CODE, and to get the correct list element for SOUND. Not all Python syntax is supported in Jinja2, but a lot is.

Variables

We can see that we've repeated range(30) a lot - it would be much better to move the 30 number into a variable to remove the hard-coding.

Replace all the occurrences of 30 with ROCKET_NUMBER, and put this line above the [cylc] section at the top of the file:

{%- set ROCKET_NUMBER = 30 %}

Variable expressions

We should also replace some of the other hard-coded numbers - the first loop should become:

{%- for num in range(30) %}
{%- set num_label = '%02d' % num %}
{%- if num <= ROCKET_NUMBER/2 or num == ROCKET_NUMBER - 1 %}
            ignite_rocket_{{ num_label }} => \
{%- else %}
                          ignite_rocket_{{ num_label }} & \
{%- endif %}
{%- endfor %}

List Variables

We can also move our SOUND list into a variable - replace the line:

            SOUND = {{ ["BANG", "whoosh"][rocket_num % 2] }}

with

            SOUND = {{ ROCKET_SOUNDS[rocket_num % 2] }}

We should also put a line at the top setting the variable, next to the {%- set ROCKET_NUMBER line:

{%- set ROCKET_SOUNDS = ["BANG", "whoosh"]

rose-suite.conf

Jinja2 variables like the ones we've created can be moved into the rose-suite.conf file for easy configurability and metadata support.

Move the variables into the rose-suite.conf file by removing the two {% set ROCKET lines and appending the following text to the rose-suite.conf file:

[jinja2:suite.rc]
ROCKET_NUMBER = 30
ROCKET_SOUNDS = ["BANG", "whoosh"]

Summary So Far

We've:

  • used for loops to collapse repeated content
  • used if blocks to control differences in content
  • centralised hardcoded settings and moved them to the rose-suite.conf file.

Examining template output

We can look at our template output by getting cylc to pre-process the jinja2 out of our suite.rc, outputting it in a cylc-readable format. We first need to register and validate it. We could use cylc register and cylc validate, but as we're (hopefully) going to run the suite in a minute, run rose suite-run -- --hold to automatically do this and start the suite in a paused state.

cylc gui, cylc view -g -j

You can use the graph view in cylc gui to examine the dependency tree before you run the suite. You can also examine the text directly - run:

cylc view -g -j $SUITE_NAME

where $SUITE_NAME is the name of the suite directory. You should get a text editor with the jinja2-and-cylc-processed suite.rc, which should be functionally identical to our starting point.

Running the suite

Click the relevant control in the open cylc gui program, then bring up your most recently used terminal window (probably the cylc view one).

You should see some output!
