Rose Advanced Tutorial: Upgrade Macro Development

Introduction

This part of the Rose user guide walks you through writing macros to upgrade application configurations to newer metadata versions.

Upgrade macros are intended to keep application configurations in sync with changes to application inputs e.g. from new code releases.

You should already be familiar with using rose app-upgrade (see the Upgrade Usage tutorial) and the concepts in the reference.

Example

In this example, we'll be upgrading a boat on a desert island.

Create a new directory called make_boat_app somewhere - e.g. in your homespace - containing a rose-app.conf file with the following content:

meta=make-boat/0.1

[namelist:materials]
paddles=1
large_trees=1

Example App Explanation

This creates an app config that configures our simple boat (a raft). It references a meta-flag (for which metadata is unlikely to already exist), made up of a category (make-boat) at a particular version (0.1). The meta flag is used by Rose to locate a configuration metadata directory.

It's important to note that the version in the meta flag doesn't have to be numeric - it could be vn0.1 or alpha or Roomy-Raft.

We need to create some metadata to make this work.

Example Metadata

Create a rose-meta/make-boat/ directory somewhere, for example in your homespace (mkdir -p ~/rose-meta/make-boat).

This is the category (also called command) directory for the metadata, which will hold sub-directories for actual configuration metadata versions (each containing a rose-meta.conf file, etc).

Usually, this would be centrally stored, under version control with the source code somewhere.

Creating Example Metadata Versions

We know we need some metadata for the 0.1 version, so create a 0.1/ subdirectory under rose-meta/make-boat/.

We'll need a rose-meta.conf file there too, so create an empty one - e.g. if the directories are in your homespace, run:

touch ~/rose-meta/make-boat/0.1/rose-meta.conf

Creating the First Metadata Version

We can safely say that our two namelist inputs are essential for the construction and testing of the boat, so we can paste the following into the newly created rose-meta.conf file:

[namelist:materials=paddles]
compulsory=true
range=1:
type=integer

[namelist:materials=large_trees]
compulsory=true
values=1

So far, we have a normal application configuration which references some metadata, somewhere, for a category at a certain version.

Let's make another version to upgrade to.

Creating the Next Metadata Version

We can now make an outrigger for our canoe, to make it more stable. Some of the inputs in our application configuration will need to change.

After upgrading, our application configuration would probably need to look a bit like this (don't change it yet!):

meta=make-boat/0.2

[namelist:materials]
paddles=1
small_trees=1
large_trees=1

It looks like we've added an input, small_trees.

Metadata for the Next Metadata Version

Let's create the new metadata version first, to document the change.

Create a new subdirectory, rose-meta/make-boat/0.2/, containing a rose-meta.conf file that looks like this:

[namelist:materials=paddles]
compulsory=true
range=1:
type=integer

[namelist:materials=small_trees]
compulsory=true
values=1,2

[namelist:materials=large_trees]
compulsory=true
values=1

We now want to automate the process of updating our app config from make-boat/0.1 to the new make-boat/0.2 version.

versions.py

Upgrade macros are invoked through a Python module, versions.py, that doesn't live with any particular version metadata - it should be present at the root of the category directory.

Create a new file rose-meta/make-boat/versions.py. You can check that everything is OK so far by changing directory to the rose-meta/make-boat/ directory and running ls -R - it should look something like:

.:
0.1  0.2  versions.py

./0.1:
rose-meta.conf

./0.2:
rose-meta.conf

Upgrade Macros Explained

Upgrade macros are Python objects with a BEFORE_TAG (e.g. "0.1") and an AFTER_TAG (e.g. "0.2").

When a user requests an upgrade for their configuration (e.g. by running rose app-upgrade), the versions.py file will be searched for a macro whose BEFORE_TAG matches the version (the 0.1 in meta=make-boat/0.1). When that particular upgrade macro is run, the version in the app configuration will be changed from BEFORE_TAG to AFTER_TAG (e.g. meta=make-boat/0.1 to meta=make-boat/0.2), as well as making other changes to the configuration if needed, like adding/removing the right variables.

Chaining Macros

If the user wanted to upgrade across multiple versions - e.g. 0.1 to 0.4 - there would need to be a chain of objects whose BEFORE_TAG was equal to the last AFTER_TAG, ending in an AFTER_TAG of 0.4.

We'll cover multiple version upgrading later in the tutorial.

Upgrade Macro Skeleton

Upgrade macros are bits of Python code that essentially look like this:

class Upgrade272to273(rose.upgrade.MacroUpgrade):

    """Upgrade from 27.2 to 27.3."""

    BEFORE_TAG = "27.2"
    AFTER_TAG = "27.3"

    def upgrade(self, config, meta_config=None):
        """Upgrade the application configuration (config)."""
        # Some code doing something to config goes here.
        return config, self.reports

They are sub-classes of a particular class, rose.upgrade.MacroUpgrade, which means that some of the Python functionality is done 'under the hood' to make things easier.

You shouldn't need to know very much Python to get most things done.

Example Upgrade Macro

Paste the following into your versions.py file:

import rose.upgrade


class MyFirstUpgradeMacro(rose.upgrade.MacroUpgrade):

    """Upgrade from 0.1 (Crafty Canoe) to 0.2 (Outrageous Outrigger)."""

    BEFORE_TAG = "0.1"
    AFTER_TAG = "0.2"

    def upgrade(self, config, meta_config=None):
        """Upgrade the boat!"""
        # Some code doing something to config goes here.
        return config, self.reports

This is already a functional upgrade macro - although it won't do anything. Note that the name of the class (MyFirstUpgradeMacro) doesn't need to be related to the versions - the only identifiers that matter are the BEFORE_TAG and the AFTER_TAG.

Upgrade Macro API

We need to get the macro to add the new 0.2 option, namelist:materials=small_trees.

We can use the nice API provided to express this in Python code - replace the # Some code doing something... line with:

        self.add_setting(config, ["namelist:materials", "small_trees"], "1")

Upgrade Macro Explanation

This changes the app configuration (config) in the way we want, and (behind the scenes) adds some things to the self.reports list mentioned in the return config, self.reports line. Note that because we were adding tree_trunks, we needed a value to add it with - what it is will often be a matter of judgement.

Running rose app-upgrade

Our upgrade macro will now work - change directory to the application directory and run:

rose app-upgrade --meta-path=path/to/rose-meta

where --meta-path equals the rose-meta/ directory you created. This option won't usually be needed by users, if the metadata is centrally stored.

This should display some information about the current and available versions - see the help by running:

rose help app-upgrade

Upgrading with rose app-upgrade

Let's upgrade to 0.2. Run:

rose app-upgrade --meta-path=path/to/rose-meta 0.2

This should provide you with a summary of changes, and prompt you to accept them. If you accept them, the app config file should change accordingly.
